## 1. 背景与目标

本项目是面向 B2B 的手套批发商城（一次性、医用、工业、园艺手套，按箱/按件销售）。当前商品为标准 SKU，缺乏"定制"能力。而在手套 B2B 场景中，**印制企业 LOGO、品牌名称、员工姓名**是高频且高毛利的需求：医院、清洁公司、汽修厂、餐饮连锁常需在手套或包装上印制自有品牌标识。

### 1.1 业务目标
- 为标准商品叠加"定制"能力，提升客单价与毛利（定制含溢价 + 一次性开版费）。
- 用可控流程承接定制订单——**配置/设计 → 提交 → 稿件/打样确认 → 报价 → 审批 → 生产**，避免"客户以为能印、实际做不了"的纠纷。
- 复用现有 B2B 能力（`company` / `quote` / `approval`），不重复造轮子。

### 1.2 衡量指标（建议）
- 定制询价单转化率：提交询价 → 成单。
- 定制订单占 GMV 比例、平均溢价率。
- 稿件/设计一次通过率（打样返工次数）。
- 从提交到打样确认的平均时长。
- 在线设计编辑器使用率与其转化率对比简易配置器。

## 2. 术语

| 术语 | 含义 |
|------|------|
| 定制项 (Customization Request) | 附加在某个可定制商品上的一组个性化配置。|
| 定制模式 (Mode) | `configurator`（简易配置器）或 `editor`（在线设计编辑器）。|
| 稿件 (Artwork) | 客户上传的 LOGO / 设计源文件（矢量或位图）。|
| 设计模板 (Design Template) | 商品上配置的可视化底稿：2D 平面图/模切图或 3D 模型 + 可印区定义。|
| 设计文档 (`design_document`) | 编辑器产出的结构化设计定义（JSON），唯一真相，可重载还原。|
| 可印区 (Print Area) | 模板上允许印制的区域，携带安全区、出血、物理尺寸、DPI、色数上限。|
| 打样 / 数码样 (Proof) | 卖方据稿件/设计生成的效果确认图，需客户签字确认。|
| 开版费 / 制版费 (Setup Fee) | 每个 LOGO / 每种颜色首次制版的一次性费用。|
| MOQ | 最小起订量（Minimum Order Quantity）；定制品有独立且更高的 MOQ。|
| RFQ / 询价单 | 现有 `quote` 模块提供的报价流程。|

## 3. 用户角色

- **采购方 / 公司买家（B2B customer / employee）**：选择可定制商品、配置或在线设计、上传 LOGO、发起询价。
- **公司审批人（approver）**：复用现有 `approval` 模块，对定制订单（金额/内容）审批。
- **卖方运营 / 客服（admin）**：后台审核稿件/设计、维护商品定制能力与设计模板、上传打样图、报价。
- **卖方设计/生产（外部/线下）**：本期不纳入系统，作为线下环节；系统只承载稿件、设计文档、打样图的传递。

## 4. 范围与两种定制模式

本功能提供**两种定制模式**，由商品是否配置了「设计模板」决定，商城端据此动态选择入口：

| 模式 | 触发条件 | 交互 | 产出 |
|------|----------|------|------|
| **① 简易配置器 (Configurator)** | 商品可定制、**无**设计模板 | 商详页内联表单：上传 LOGO / 输入文字 / 选位置 / 选颜色 | 定制项 + 稿件文件 |
| **② 在线设计编辑器 (Editor)** | 商品可定制、**有**设计模板 | 商详页"在线设计"入口 → 跳转独立编辑页，可视化摆放 | 定制项 + `design_document` + 预览图 + 资产 |

两种模式**共用同一数据流**（同一 `customization_request` 表、同一询价/审批流程），仅编辑体验不同。

### 4.1 本期范围（In Scope）
1. 商品级"是否可定制"开关及定制能力配置（可印区、方式、颜色数上限、MOQ、开版费、溢价）。
2. **简易配置器**：方式选择、LOGO 上传、文字输入、位置/颜色选择，前后端实时校验。
3. **在线设计编辑器（2D 编辑 + 3D 预览）**：见 §7 架构。P1.5 交付 2D 编辑 + 可选 3D 预览视口。
4. 商品设计模板配置（模板资产 + 可印区/安全区/物理尺寸/DPI）。
5. 稿件、设计文档、预览图存储（走 Medusa File 模块）。
6. 定制信息随行项（line item）进入购物车 / 询价单，后台完整可见。
7. 打样确认流程：admin 上传打样图 → 客户在订单/询价详情页确认或驳回。
8. 定价：开版费 + 单价溢价，通过现有 `quote` 报价给出最终价。
9. 与现有 `approval` 审批流打通。

### 4.2 非本期范围（Out of Scope）
- **在 3D 空间内自由编辑/摆放**（本期 3D 仅为只读预览；编辑始终在 2D 层，见 §7）。
- 自动去底 / 矢量化 / AI 生成图案等图像处理。
- 自助结账直接下单定制品（本期定制品一律走询价，不进入即时结算）。
- 生产排产、供应商协同系统对接。

## 5. 功能需求

每个可定制商品由 admin 配置支持哪些维度，两种模式均据此约束。

### 5.1 定制方式 (Method)
- `logo`：印制 LOGO 图案；`text`：印制文字（企业名/员工名/部门/编号）；`both`：组合。
- 印制工艺枚举（影响价格/交期，本期做数据字段）：丝印 (screen-print)、热转印 (heat-transfer)、包装盒印刷 (box-print) 等。

### 5.2 LOGO / 稿件 (FR-LOGO)
- 上传格式：`svg` `pdf` `ai` `eps`（矢量优先），`png` `jpg`（位图，需最小分辨率提示）。
- 单文件大小上限（建议 20MB，配置化）。
- 上传后展示缩略图与文件名，可删除重传。
- 记录原始文件名、MIME、尺寸/分辨率（能取到时）。

### 5.3 文字 (FR-TEXT)
- 单行/多行，字符数上限（配置化，建议每行 ≤ 24）。
- 字体白名单、颜色（受色数上限约束）、字号、行距。
- 实时字符计数与非法字符校验。

### 5.4 印制位置 / 可印区 (FR-POS)
- 位置枚举（配置化，按商品类型不同）：手背 (back-of-hand)、袖口 (cuff)、包装盒 (box/case)；每只手套 (per-glove) vs 仅外箱 (case-only)。
- 编辑器模式下每个位置对应模板上一个**可印区 (Print Area)**，可分别放置内容。

### 5.5 颜色 (FR-COLOR)
- 支持颜色数上限（如"最多 2 色"），超出提示升级工艺或转人工报价。
- 可提供 Pantone 色号输入（自由文本 + 校验提示）。

### 5.6 数量与 MOQ (FR-MOQ)
- 定制品有独立 MOQ（通常远高于标准品），低于 MOQ 无法提交并提示差额。
- 数量按现有"箱/件"单位承载，与 variant（尺寸）组合。
- 可跨尺寸拼单满足 MOQ（进阶，可后置）。

### 5.7 备注 (FR-NOTE)
- 自由文本备注（特殊要求、交期期望等）。

### 5.8 在线设计编辑器 (FR-EDITOR)
当商品配置了设计模板时启用；否则回退简易配置器。能力（MVP）：
- **元素**：添加/删除 LOGO 图片（上传，见 §5.2）与文字（见 §5.3）。
- **变换**：拖拽、等比/自由缩放、旋转、层级 (z-order)、对齐/居中辅助线。
- **约束**：元素限制在可印安全区内；越界或低于目标 DPI 实时告警。
- **多区域**：可在不同可印区分别放置内容。
- **3D 预览（若模板为 3D）**：实时把当前 2D 设计包裹到模型上，可旋转查看（只读，见 §7）。
- **草稿**：登录用户可保存/命名/续编设计草稿。
- **产出**：① `design_document`（JSON，见 §7.3）；② 预览图（PNG）；③ 关联上传资产 `file_id`。
- **可复现**：`design_document` 能被编辑器重载还原并再次编辑生成新版本。

## 6. 用户流程

### 6.1 简易配置器流程（无模板商品）
1. 商详页 `is_customizable=true` 且无模板 → 展示内联"定制此商品"。
2. 选择方式 → 上传 LOGO / 输入文字 → 选位置/颜色 → 选尺寸与数量。
3. 前端实时校验：MOQ、色数、字符数、文件格式/大小、必填。
4. 展示价格构成说明："最终价格以卖方报价为准，含一次性开版费 + 单价溢价"。
5. 加入购物车（携带定制元数据的行项）。
6. 结算入口对定制品替换为"提交询价 (Request Quote)"，进入 `quote` 流程；启用审批则触发 `approval`。

### 6.2 在线设计编辑器流程（模板商品）
1. 商详页有设计模板 → 展示"在线设计 (Design Online)"入口。
2. 点击**跳转独立编辑页**（新路由，非弹窗），加载模板与可印区。
3. 添加 LOGO/文字，拖拽/缩放/旋转；实时约束可印区、越界/低 DPI 告警；若为 3D 模板可切到 3D 视口旋转查看效果。
4. 可**保存草稿**（命名、续编）；`design_document` 持久化并可重载。
5. "用此设计下单"：选尺寸/数量 → 校验 MOQ → 导出预览图 → 生成/更新 `customization_request`（携带 `design_document` 与预览图）→ 加入购物车。
6. 同 §6.1 第 6 步：结算导向询价，触发审批。
7. 购物车/询价/后台展示预览图，并提供"再次编辑"回到编辑器（生成新版本）。

> 说明：编辑器产出属"客户自助初稿"，仍进入 §6.3 打样确认——卖方据 `design_document` + 资产出正式打样图供客户最终签字，规避屏幕与实际印制的色差/工艺偏差纠纷。

### 6.3 打样确认（提交后）
1. Admin 审核稿件/设计，上传打样图 (proof)，状态置"待客户确认"。
2. 客户在询价/订单详情页看到打样图，可**确认**或**驳回并留言**。
3. 确认后进入报价/生产；驳回回到 admin 重处理，保留版本历史。

### 6.4 报价与审批
- 复用 `quote`：admin 填入开版费、单价（含溢价）、交期，客户接受转订单。
- 复用 `approval`：金额/内容触发公司内部审批。

### 6.5 定制项状态机
```
draft → submitted → artwork_review → proof_pending → proof_approved → quoted → accepted → in_production
                          ▲                   │
                          └──── rejected ──────┘   (驳回回到 artwork_review)
```

## 7. 编辑器架构决策：2D/3D 统一分层

> **核心决策：2D 编辑器与 3D 预览不是两套技术栈，而是"一个系统、一份数据、分层渲染"。** 这把 3D 从"高风险大功能"降级为"给 2D 编辑器加一个只读视口"。

### 7.1 图形学前提
手套印制的"编辑"本质是 **2D** 的——用户在一块平面可印区里摆 LOGO 与文字，3D 只是把这块 2D 图案按 **UV** 包裹到几何体上做效果预览。用户不需要在三维自由空间里编辑。因此编辑面天然是 2D，3D 是可视化。

### 7.2 三层分层

```
① 数据层   design_document（唯一真相）
            = 各可印区内的 2D 元素 + 变换（mm / 归一化坐标）
                        │
        ┌───────────────┴────────────────┐
        ▼                                ▼
② 编辑层  2D 画布 (Konva)           ③ 预览层  3D 视口 (R3F / Three.js)
   真正的创作工具                    只读渲染：把 ② 的画布当纹理
   变换手柄 / 吸附 / 文字 / 层级       CanvasTexture → UV 贴到模型
```

**决定性的一招**：Konva 画布导出的就是一个 `HTMLCanvasElement`，直接作为 Three.js 的 `CanvasTexture` 纹理源。于是 Konva 不是"被 3D 重造一遍的另一个编辑器"，它**本身就是 3D 模型的纹理源**；二者是叠加，不是重复。

因此"要紧的东西"都只有一份：一份 `design_document`、一套可印区/安全区/DPI 校验、一套 React 组件与状态。3D 只是"多挂一个视口"。

### 7.3 技术选型

| 职责 | 选型 | 理由 |
|------|------|------|
| 2D 创作 | **Konva** | 变换手柄、吸附、命中检测、文字编辑开箱即用 |
| 3D 预览 | **React Three Fiber + Three.js + @react-three/drei** | React 19/Next 15 生态，声明式，`useGLTF` 载模型、`OrbitControls` 转视角、`Environment` 打光 |
| 3D 文字（如需 3D 内直接排文字） | **troika-three-text** | SDF 渲染，任意缩放不糊 |
| 2D↔3D 纽带 | **CanvasTexture** | Konva canvas → Three 纹理，随编辑更新 |
| 模型格式 | **glTF / GLB + Draco 压缩** | Web 3D 标准，压体积 |
| 编辑器局部状态 | **zustand** | R3F 生态标配；高频编辑状态不入 React Context（避免全树重渲染） |

**不采用**：
- 纯 Three.js 画 2D（正交相机）——要在 3D 引擎里手搓 2D 编辑器的手柄/吸附/文字，得不偿失。
- 在 3D 里自由拖贴花编辑 (decal)——独立且更难的交互栈，对平面可印区 UX 更差（要跟透视/遮挡/曲面较劲）。仅当设计需绕缠/跨面、无法归约成平面可印区时才考虑。
- Babylon.js / Threekit / Zakeke 等——引擎过重或 SaaS 锁定，难与 Medusa 询价/审批数据流深度打通。

### 7.4 `design_document` schema（草案）
以**可印区相对坐标（归一化 0–1 或 mm）**记录，不绑定像素分辨率，便于换模板/生产出图：
```jsonc
{
  "version": 1,
  "unit": "normalized",           // normalized | mm
  "areas": [
    {
      "areaId": "back-of-hand",
      "elements": [
        { "type": "image", "fileId": "file_123", "x": 0.3, "y": 0.2,
          "w": 0.4, "h": 0.25, "rotation": 0 },
        { "type": "text", "content": "ACME", "font": "inter-bold",
          "colorRef": "c1", "x": 0.3, "y": 0.5, "w": 0.4, "rotation": 0 }
      ]
    }
  ],
  "colors": { "c1": "#0A3D62" }    // 受色数上限约束
}
```

### 7.5 三个必须落实的图形学细节
1. **分辨率/DPI**：纹理按印制 DPI 出图（可印区 mm × DPI），通常大于屏幕编辑视口。做法：维护**离屏高分辨率 Konva stage**当纹理源，屏幕编辑视图缩放显示。
2. **色彩管理**：canvas 是 sRGB，Three 里须设 `texture.colorSpace = SRGBColorSpace`，否则 3D 预览偏色、与 2D 对不上。
3. **模型 UV 是硬约束**：可印区在模型上必须是**低畸变的干净 UV 矩形**——这是给 3D 建模方的交付规范，建模时把可印区 UV 铺平，否则贴图拉伸变形。

### 7.6 Next.js 集成注意
- R3F `<Canvas>` 必须 `"use client"`，且在 App Router 里 `dynamic(() => import(...), { ssr: false })`——WebGL 依赖 DOM，不可 SSR。
- 导出预览图：`gl.domElement.toDataURL()` 截取视角或离屏渲染固定角度，接入 `preview_file_id` 存储。
- GLB 模型走 File 模块 / CDN，配 Draco 解码器路径。

## 8. 后端数据模型与架构

遵循 CLAUDE.md：改动 data model / link 需 `db:migrate`；模块 CRUD 由 `MedusaService` 自动生成；模块键常量放各模块 `index.ts`；避免硬编码字符串。

### 8.1 方案选择
- **方案 A（最小改动）**：定制信息作为 line item `metadata`（JSON）+ 稿件走 File 模块。优点无迁移；缺点难结构化查询、版本/审计弱、admin 检索差。
- **方案 B（推荐）**：新建 `customization` 模块，结构化存储定制项/稿件/设计/打样，用 module link 关联 line item / order / quote / company。

### 8.2 推荐模块：`customization`
目录 `backend/src/modules/customization/`：`index.ts`（导出 `CUSTOMIZATION_MODULE` 常量 + `Module()`）、`service.ts`（`extends MedusaService({...})`）、`models/`、`migrations/`。

模型（`model.define`）：
- **`customization_request`**（一次定制配置）
  - `id`、`mode`（`configurator` / `editor`）、`method`（`logo`/`text`/`both`）、`print_technique`（可空）
  - `positions`（json，配置器模式）、`text_content`（json）、`color_count`、`pantone`（可空）
  - `moq`、`quantity`、`status`（见 §6.5）、`notes`
- **`customization_artwork`**（稿件，1:N，含版本）
  - `id`、`file_id`、`original_filename`、`mime`、`width`/`height`（可空）、`version`、`is_current`
- **`customization_design`**（编辑器产出，1:N，含版本；仅 `mode=editor`）
  - `id`、`template_id`、`design_document`（json，见 §7.4）、`preview_file_id`、`version`、`is_current`、`is_draft`
- **`customization_proof`**（打样 + 客户确认）
  - `id`、`file_id`、`version`、`status`（`pending`/`approved`/`rejected`）、`customer_comment`、`reviewed_by`

> **设计模板 (Design Template)** 承载在商品侧，字段：`template_type`（`flat_2d`/`dieline`/`model_3d`）、`base_asset_file_id`、`print_areas`（json：几何范围、`safe_zone`、`bleed`、物理尺寸 mm、`dpi`、每区 `methods`/`max_colors`）。建议随商品扩展存储以随商品生命周期管理。

### 8.3 Module Links（`backend/src/links/`）
- `line-item-customization`：购物车/订单行项 ↔ `customization_request`（核心承载）。
- `quote-customization`：`quote` ↔ `customization_request`（报价上下文）。
- `company-customization`（可选）：按公司检索历史定制。

### 8.4 Workflows（`backend/src/workflows/customization/`）
- `create-customization-request`：校验（MOQ/色数/文件/可印区约束）→ 落库 → link 到 line item。
- `save-design`（编辑器）：校验 `design_document` 元素落在可印区、DPI 达标 → 存设计版本 + 预览图。
- `submit-artwork` / `upload-proof` / `respond-to-proof`：状态流转 + 补偿。
- 报价/审批复用现有 quote / approval workflow，经 hook 注入定制上下文。

### 8.5 API 路由（`backend/src/api/`）
- `store/customizations`：定制项 CRUD、稿件上传、设计草稿保存/加载、打样确认/驳回。
- `store/products/:id/design-template`（或并入商品字段）：拉取模板供编辑器初始化。
- `admin/customizations`：审核稿件/设计、上传打样、检索。
- `admin/products/:id/design-template`：配置模板资产与可印区。
- 每路由配套 `validators.ts`（zod，兼容 3.22.4）、`query-config.ts`、`middlewares.ts`。

### 8.6 存储
- 稿件、设计源文件、预览图、GLB 模型统一走 Medusa **File 模块**；DB 仅存 `file_id` 与元数据。
- 源文件（.ai/.eps/.glb）不做浏览器内编辑预览，仅供下载/加载；提供缩略图/预览图字段（可空）。

## 9. 商城端实现要点（storefront）

遵循 CLAUDE.md：后端调用集中在 `src/lib/data/*.ts`（`"use server"` + SDK + cache tags）；UI 放 `src/modules/<feature>/`；常量放 `src/lib/constants.tsx` / `config.ts`。

- 新增 `src/lib/data/customization.ts`：定制项 CRUD、稿件上传、设计草稿保存/加载、打样确认的 server actions。
- 新增 `src/modules/customization/`：
  - 简易配置器组件（方式/上传/文字/位置/颜色/数量校验）。
  - **在线设计编辑器**：`"use client"` 客户端组件；2D 用 Konva，3D 预览用 R3F（`dynamic` + `ssr:false`）；状态用 zustand。
  - 打样确认组件。
- 商详页（`src/modules/products/`）按 `is_customizable` 与**是否有设计模板**渲染入口：无模板 → 内联配置器；有模板 → "在线设计"入口。

## 10. Admin 端要点（backend `src/admin/`）

- 商品编辑页扩展：**定制能力面板**（开关、方式、位置、色数上限、开版费、溢价、MOQ）。
- **设计模板配置面板**：上传底图/模切图/GLB、圈定可印区/安全区/出血、设置物理尺寸与目标 DPI、每区色数上限；3D 模板需校验 UV 铺平规范。
- 订单/询价详情页扩展：展示完整定制信息、**设计预览图与 `design_document`**、下载稿件/设计源文件、上传打样图、查看确认状态与历史版本。

## 11. 定价模型

最终价 = **一次性开版费（每 LOGO / 每色）** + **（标准单价 + 单价溢价）× 数量**。
- 开版费、溢价由 admin 在商品定制配置中设默认值，报价时可覆盖。
- 定制品**不走即时结算**，最终价以 `quote` 报价为准（承接工艺/色数/交期不确定性）。
- 溢价支持"绝对值"或"百分比"（配置化）。

## 12. 非功能需求

- **校验一致性**：前端 + 后端（zod）双重校验，MOQ/色数/文件格式/可印区以后端为准。
- **文件安全**：校验 MIME 与扩展名，限大小，杜绝可执行文件；下载走鉴权。
- **权限**：仅归属公司成员可见/编辑本公司定制项与草稿；打样确认需登录且属该公司。
- **可审计**：稿件、设计、打样均带版本与操作人，状态流转可追溯。
- **国际化**：文案随 `[countryCode]` i18n；价格随 region 币种（`eur`/`usd`）。
- **性能**：大文件异步上传带进度；配置器交互 < 100ms；编辑器画布拖拽/缩放目标 60fps，模型/底图懒加载。
- **兼容与降级**：编辑器桌面优先；不支持环境（低端移动端、无 WebGL）回退简易配置器。
- **数据可复现**：`design_document` 与坐标系版本化；模板资产更新需保证旧设计可加载或明确失效提示。

## 13. 边界与异常场景

- 上传格式/尺寸不符 → 明确提示与建议（矢量优先、最小分辨率）。
- 数量低于 MOQ → 阻止提交并提示差额。
- 色数超上限 → 提示升级工艺或转人工报价。
- 编辑器元素越出可印安全区 → 阻止保存或强制回拉并提示。
- 位图 LOGO 低于目标 DPI → 告警"清晰度不足，可能影响印制"，允许继续但标注风险。
- 打样被驳回多次 → 保留全部版本，客服可介入。
- 商品设计模板被更新/删除 → 保留历史设计快照，编辑时提示模板已变更。
- 定制品与标准品混在同一购物车 → 结算拆分（标准品即时下单，定制品走询价）或整单转询价（见 §15 决策）。
- 询价过程中客户修改稿件/设计 → 生成新版本，旧报价失效需重报。
- 草稿归属仅本人/本公司可见。

## 14. 交付阶段

- **P0 · MVP**：商品可定制开关 + 简易配置器（上传/文字/位置/颜色/数量校验）+ 定制项随行项进询价 + admin 可见并下载稿件 + 复用 quote 报价与 approval 审批。
- **P1**：打样确认流程（proof 上传 + 确认/驳回 + 版本）、开版费/溢价自动带入报价、admin 定制配置面板。
- **P1.5 · 在线设计编辑器**：设计模板配置（底图/可印区/DPI）+ 独立编辑页（Konva 2D：摆放/变换/区域约束/DPI 校验）+ `design_document`/预览图落库 + 草稿保存 + "再次编辑"。**3D 模板启用 R3F 只读预览视口**（复用同一 `design_document`，见 §7）。
- **P2**：多可印区多内容、跨尺寸拼单满足 MOQ、Pantone 校验、字体库扩展、定制历史与复购。
- **P3（探索）**：3D 内自由摆放/贴花编辑、稿件自动检测（分辨率/去底/矢量化建议）、AR 预览。

## 15. 待决策问题（Open Questions）

1. 定制品是否**一律走询价**，还是允许"简单文字定制即时下单"？—— 影响结算拆分。
2. 开版费/溢价是**平台统一**还是**按公司协议价**（结合 `company-customer-group`）？
3. 印制工艺、位置、字体白名单的**初始枚举**由业务方提供。
4. 定制品 MOQ 的**默认值与单位**（按只 / 按箱）。
5. 稿件/设计的**保密与保留期限**（合规/隐私）要求？
6. 混合购物车（定制 + 标准）如何拆单？
7. 打样是否需**卖方线下人工**参与，还是要求数字化打样图？
8. 在线编辑器首批支持哪些**模板类型**（2D 平面图 / 模切图 / 3D）？3D 预览是否列入 P1.5 首发。
9. 设计模板资产（底图/可印区/DPI/GLB + UV 规范）由谁生产——卖方设计团队线下制作上传，还是需标准化制作规范？
10. 编辑器**字体白名单**，是否允许客户上传自有字体（字体版权）。
11. 编辑器初稿是否**仍强制走打样确认**，还是简单场景可跳过直接生产？

### 16.2 P1.5（在线设计编辑器）
- [ ] Admin 可为商品上传设计模板（底图/可印区/安全区/物理尺寸/DPI；3D 含 GLB + UV 校验）。
